# E2EE Implementation Plan - ChatApps_Pigeons

## 1. Overview

This document outlines the design and implementation steps for End-to-End Encryption (E2EE) in the ChatApps_Pigeons platform. The goal is to ensure that only the sender and intended recipients can read message contents, protecting data from server-side access.

The system uses a **Unified Random Key Model**: both 1-1 and group conversations follow the exact same key distribution flow — a `shared_key` (AES-256) is generated by the initiator, encrypted with each participant's RSA public key, and stored in `ConversationKeysVault`. 

## 2. Cryptographic Specifications

- **Identity Key Pair**: RSA-OAEP (2048-bit) — used to **wrap/unwrap `shared_key` vault entries** via `crypto.subtle.wrapKey()` / `crypto.subtle.unwrapKey()`.
- **Conversation Key (`shared_key`)**: AES-GCM 256-bit, randomly generated per conversation (or per rotation). (IV: 12 bytes).
- **Key Derivation (for PIN)**: PBKDF2 with **SHA-256**, 100,000 iterations → produces `KEK` (Key Encryption Key). (Salt: 16 bytes).
- **Data Formats**: RSA public keys in **SPKI** (Base64); encrypted private key in **JWK** wrapped by KEK.


## 3. Key Management

### 3.1 Local Storage (Client-side via Dexie.js)

To ensure absolute data isolation between different users on the same device, we use a **Dynamic Database Model**:

- **Database Name**: `pigeons_db_{user_id}` (e.g., `pigeons_db_123`).
- Each user has their own isolated IndexedDB instance.

The schema for each user's database is as follows:

- **Table: `own_keys`** (Stores current user's identity keys)
    - `user_id`: Primary Key
    - `public_key_spki`: RSA-OAEP public key, SPKI Base64 string
    - `private_key_obj`: RSA-OAEP private key, CryptoKey (extractable: true)
- **Table: `conversation_keys`** (Stores shared AES keys per conversation per version)
    - `conversation_id`: Foreign Key (Index)
    - `shared_key`: CryptoKey object (AES-GCM 256-bit, extractable: true)
    - `key_version`: Primary Key (Integer)
- **Table: `messages`** (Standard message table with E2EE fields)
    - `id`: Primary Key (from server)
    - `local_id`: Temporary ID for optimistic UI
    - `conversation_id`: Index for fast lookup
    - `sender_id`: User ID
    - `content`: **Plain text** (decrypted locally, never sent to server)
    - `is_e2ee`: Boolean flag
    - `key_version`: Reference to the `shared_key` used for this message
    - `created_at`: Timestamp (Index for ordering)

- **In-memory Cache**:
    - `conversation_keys`: A Map of `(conversation_id + key_version) -> shared_key` object. This allows instant decryption of both new messages and history without re-fetching from IndexedDB.

### 3.2 Server-side Storage (Database)

The following tables are required on the server:

- **`Users` Table Updates**:
    - `public_key`: RSA-OAEP public key, SPKI Base64 string.
    - `wrapped_private_key`: The RSA private key wrapped with KEK (AES-GCM).
    - `kek_iv`: Base64 string (The 12-byte random IV used for the private key).
    - `pin_salt`: Base64 string, Random salt for PBKDF2 (16 bytes).
- **`Messages` Table Updates**:
    - `is_e2ee`: Boolean.
    - `key_version`: Integer (the version of `shared_key` used to encrypt this message).
    - `iv`: Base64 string (random 12-byte Initialization Vector per message).
- **`ConversationKeysVault` Table**:
    - `id`: Primary Key.
    - `user_id`: The owner of this vault entry.
    - `conversation_id`: Foreign Key.
    - `key_version`: Integer.
    - `wrapped_shared_key`: The `shared_key` wrapped with the **owner's RSA-OAEP public key**.
- **Server-side Security**:
    - **PIN Rate Limiting**: Backend must implement account lock after 5 consecutive failed PIN attempts to prevent brute-force attacks.

## 4. Workflows

### Phase 1: Setup & Recovery

1. **Initial Setup**:
    - User enters a 6-digit **PIN**.
    - Client generates a random `pin_salt`.
    - Client derives `KEK` (Key Encryption Key) from PIN + `pin_salt` via PBKDF2.
    - Client generates RSA-OAEP key pair.
    - Client generates `kek_iv` 12 bytes
    - Client wraps the private key with `KEK` + `kek_iv` → `wrapped_private_key`.
    - Client uploads `public_key`, `wrapped_private_key`, `kek_iv` and `pin_salt` to server.
2. **Recovery (New Device)**:
    - User logs in and enters PIN.
    - Client fetches `wrapped_private_key`, `kek_iv`, and `pin_salt` from server.
    - Client derives `KEK` from PIN + `pin_salt`.
    - Client unwraps private key using `KEK` + `kek_iv` and stores it in `IndexedDB`.
    - **Bulk Sync**: Client fetches the entire `ConversationKeysVault`.
    - Client unwraps all `wrapped_shared_key` entries using its private key and populates the local `conversation_keys` table to enable immediate message preview and history access.
3. **PIN Change**:
    - Unwrap private key with old KEK → Re-wrap with new KEK → Upload to server.

### Phase 2: Real-time Messaging & Sync

1. **Real-time (Socket)**:
    - Messages are emitted to `conversationId` room.
    - Active devices receive and decrypt immediately using the cached `shared_key`.
2. **Reliability (Sync API)**:
    - On connection/app-start, client calls API to fetch unread/missed messages.
    - Server returns missed encrypted messages.
    - Client decrypts each message using the `shared_key` for its `key_version` and fills gaps in `IndexedDB`.

### Phase 3: Unified Key Management (1-1 & Group)

Both 1-1 and group conversations follow the **identical flow**:

- **For 1-1**: The first sender generates the `shared_key`.
- **For Group**: The admin generates the `shared_key` upon group creation.

1. **Key Generation & Distribution**:
    - Initiating client generates a random `shared_key` (AES-GCM 256-bit).
    - For **each participant** (including self), fetch their `public_key` from server, then:
        ```
        wrapped_shared_key = wrapKey(shared_key, participant_public_key, "RSA-OAEP")
        ```
    - Upload `{ user_id, conversation_id, key_version, wrapped_shared_key }` to `ConversationKeysVault` for each participant.

2. **Sending a Message**:
    - Retrieve `shared_key` from in-memory cache (or IndexedDB fallback).
    - Generate a random `iv`.
    - Encrypt plaintext: `AES-GCM(shared_key, iv, plaintext)` → `encrypted_content`.
    - Emit: `{ encrypted_content, iv, key_version, conversation_id }`.
    - Store plain text in local `IndexedDB`.

3. **Receiving a Message**:
    - Look up `shared_key` by `conversation_id` + `key_version` from cache/IndexedDB.
    - If not found: fetch from `ConversationKeysVault`, unwrap with own private key, cache it.
    - Decrypt using `iv` → store plain text in `IndexedDB`.

4. **Key Rotation (Winner-takes-all)**:
    - Triggered when: a member leaves, or a new member joins with `allow_history_view = OFF`.
    - The first active client to detect the trigger generates `shared_key` at `key_version + 1` and distributes it to all current members via uploads to vault.
    - Server only accepts a new vault entry if `key_version == current_version + 1`.
    - Clients that lose the race must fetch the winning key from the vault.

5. **Group `allow_history_view` Setting**:
    - **ON**: New member requires Admin approval. Upon approval, Admin wraps the all `shared_key` with the new member's public key and uploads to vault.
    - **OFF**: Key Rotation is triggered immediately upon a new member joining, so they cannot read old messages. (requires Admin approval is optional)

## 5. Multi-device Synchronization

### 5.1 Shared Identity Model

Since the private key is backed up via PIN and synced to all devices, every device of a user shares the same RSA-OAEP key pair. This means any device can unwrap vault entries independently without coordinating with other devices.

### 5.2 Message Sync Flow

- **Encryption**: Sender encrypts the message once using the conversation's `shared_key`.
- **Distribution**: Server broadcasts the encrypted message to the **Conversation Room** (`conversationId`). All active devices of all participants join this room.
- **Decryption**: Each device retrieves the `shared_key` (by `conversation_id` + `key_version`) from local cache/IndexedDB and decrypts the message for display.
- **Outgoing Sync**: A message sent from a Desktop client appears instantly on the Mobile client as a "sent" message, since both devices share the same key pair and are in the same room.

## 6. Decision Log

- **2026-05-03**: Selected Basic Public Key model over Signal Protocol for v1 to reduce implementation time.
- **2026-05-03**: Decided to store decrypted messages locally in IndexedDB for performance/searchability.
- **2026-05-03**: Implemented PIN-based cloud backup for private keys to support multi-device usage.
- **2026-05-03**: Confirmed Shared Identity Model for multi-device sync (all devices share the same key pair via PIN recovery).
- **2026-05-03**: **Added `key_version`** to both `messages` and `ConversationKeysVault` to support history decryption across key rotations.
- **2026-05-03**: **Adopted "Winner-takes-all" mechanism** for Key Rotation: the first client to successfully upload `version + 1` defines the new `shared_key`.
- **2026-05-03**: **Mandated Server-side Rate Limiting** for PIN attempts (lock after 5 failures).
- **2026-05-03**: **Confirmed Bulk Sync strategy**: all vault keys are downloaded and unwrapped at device setup for best UX (message previews).
- **2026-05-03**: **Assumed UX over Entropy**: accepted 6-digit PIN trade-off, relying on robust server-side rate limiting.
- **2026-05-03**: **Dropped ECDH Key Agreement** in favor of Unified Random Key Model. ECDH's zero-server-knowledge property was already negated by the vault. Pure ECDH could still support multi-device (devices share the same key pair and recompute independently), but group keys still require the vault — creating two divergent code paths with no benefit.
- **2026-05-03**: **Chose RSA-OAEP over EC P-256** for identity key pair. EC P-256 keys in Web Crypto cannot directly encrypt data (only `deriveBits`/`deriveKey`), so direct vault key wrapping requires RSA-OAEP (`wrapKey(shared_key, rsa_public_key, "RSA-OAEP")`). This eliminates the need for ECIES and any ephemeral key fields in the vault schema.
- **2026-05-04**: **PIN Lockout Strategy**: Decided on temporary time-based backoff (e.g., lock for 15 mins) after 5 failed attempts, rather than permanent data loss, balancing security with UX.
- **2026-05-05**: **Key Extractability**: Updated both RSA Private Key and AES Shared Key to `extractable: true`. This is technically required for the `wrapKey()` operation to function when syncing keys to the server vault or sharing conversation keys with participants.
- **2026-05-04**: **Race Condition Mitigation**: Handled via Server-side validation (Server rejects invalid `key_version` uploads) and Client-side Fail-Soft (UI gracefully displays "cannot decrypt" without crashing).
- **2026-05-04**: **Forward Secrecy**: Opted out of time/message-based auto-rotation for simplicity. Keys only rotate on membership changes.
- **2026-05-04**: **Zero-Trust Signatures**: Deferred digital signatures for key distribution to v2. V1 will rely on Server API security (JWT/Role validation) to prevent unauthorized vault uploads.
